# Bark voice-line playback integration into the running game

The change adds real-time voice-line playback to the vertical slice by wiring a `BarkAudioPlayer` (backed by platform-specific `AudioPlayer` implementations) into `SliceDirector.fireBark()`, then enriching `CombatEngine` with contextual bark triggers (taunts, damage reactions, deaths, victories, healing) and layering exploration/idle barks over the `SliceScreen` exploration phases. A `Heal` combat action and a 30-second idle-bark timer round out the feature.

Watch for: Combat barks routed back through `fireBark` are subject to per-key cooldowns, meaning rapid combat sequences can silently swallow expected barks (confirmed). The idle bark timer uses `Random.nextInt` without a seed, making its selection untestable and non-reproducible (confirmed). On Android, every bark playback writes a temp file to disk, introducing latency spikes on low-end devices (confirmed, documented as TODO).

## High-level view

Audio playback follows a clean expect/actual factory pattern: a common `AudioPlayer` interface, desktop impl via `javax.sound.sampled.Clip`, Android impl via `MediaPlayer` + temp files. `BarkAudioPlayer` wraps this with bark-event resolution and interruption semantics — only one bark plays at a time, new barks stop the old one. This is the non-overlapping guarantee.

`CombatEngine` deterministically selects barks via an injected `Random`, enabling seeded tests for taunt selection, victory barks, and the 40% damage-reaction probability. The engine itself doesn't enforce cooldowns — it emits `BarkTriggered` events which `SliceDirector.handleCombatEvents` feeds back through `fireBark()`, where the `BarkEventBus` cooldown check lives. This means rapid same-key barks within a combat tick sequence can be silently dropped.

The `SliceScreen` wires `BarkAudioPlayer` into composition via `DisposableEffect`, with cleanup on dispose. Exploration barks fire from scene callbacks (sewer entry, enemy encounter, boss discovery) and idle barks from a coroutine that polls every second and triggers after 30 seconds of inactivity.

Test coverage focuses on combat bark triggers (14 tests) and `BarkAudioPlayer` unit tests. Exploration barks, idle barks, and the combat-bark-cooldown interaction are not tested.

<details>
<summary>Issues (5)</summary>

1. **Combat barks swallowed by cooldown** — `handleCombatEvents` routes `BarkTriggered` events back through `fireBark()`, which checks the per-key cooldown. If the same bark fires twice in rapid succession (e.g. `BRUGG_I_DONT_HAVE_MUCH_LEFT` used for both "low HP" damage and death), the second invocation will be suppressed. Consider either bypassing cooldown for combat-origin barks or ensuring the bark sets don't overlap within a single tick's events.

2. **Victory bark reuses taunt bark keys** — `victoriesFor("nib")` returns `NIB_IS_THAT_ALL_YOUVE_GOT`, which is also in `tauntsFor("nib")`. If the taunt fired on tick 1 and victory occurs within the cooldown window, the victory bark is silently dropped by the bus.

3. **Idle bark selection is non-deterministic** — `pickIdleBark` uses `Random.nextInt` (global RNG) rather than an injected `Random` instance. This makes idle bark behavior untestable and unreproducible, inconsistent with the seeded-random pattern used in `CombatEngine`.

4. **Android temp-file latency** — Every `AndroidAudioPlayer.play()` call copies the entire WAV into a temp file before playback. For bark voice lines (short clips), this adds measurable I/O overhead on each trigger. The code documents a streaming alternative via `AssetFileDescriptor` but hasn't implemented it.

5. **No exploration or idle bark tests** — The exploration barks (sewer entry, enemy encounter, boss discovery) and idle bark timer are exercised only through manual play. There are no unit or integration tests verifying their trigger conditions, cooldown interaction, or timer behavior.

## Combat bark cooldown interaction

When `CombatEngine.tick()` produces `BarkTriggered` events, `SliceDirector.handleCombatEvents` calls `fireBark()` for each one — the same path as player-initiated barks, including the `BarkEventBus` cooldown check. The engine picks barks unconditionally based on game state; the bus silently suppresses duplicates.

The problematic case: `deathBarkFor("brugg")` returns `BRUGG_I_DONT_HAVE_MUCH_LEFT`, which is also the low-HP damage bark in `damageBarkFor`. If Brugg took damage earlier in the fight (triggering `I_DONT_HAVE_MUCH_LEFT` from the low-HP path), then dies on a subsequent tick within the cooldown window, the death bark is silently swallowed.

Similarly, `victoriesFor` reuses bark keys from the taunt set:

```
tick 1: taunt fires -> NIB_IS_THAT_ALL_YOUVE_GOT -> bus records timestamp
tick 2: attack kills enemy -> victory -> pickVictoryBark -> NIB_IS_THAT_ALL_YOUVE_GOT
  -> fireBark -> BarkEventBus -> OnCooldown -> bark swallowed
```

The fix is either: (a) make victory/death bark keys distinct from taunt/damage keys, (b) give combat-origin barks a bypass flag on `fireBark`, or (c) use a zero-cooldown category for combat barks.

## Double-bark audio interruption on sewer entry

The cellar door trigger fires two barks in immediate succession: `NIB_SMELL_TREASURE` followed by a randomly-chosen exploration bark. Both pass cooldown checks (different keys), but the second `playBark` call interrupts the first at the audio layer. The first bark produces a Questbook entry but is never heard. If both should be audible, a queue or delay is needed between the two `fireAndFlash` calls.

## Healing bark cooldown safety

Healing bark keys (`NIB_GOOD_AS_NEW`, `BRUGG_I_FEEL_BETTER_THAN_EVER`, `VELLUM_IM_BACK_ON_MY_FEET`) are distinct from all other bark categories, so they won't collide with taunt/damage cooldowns. This is the right approach — contrast with the victory/taunt key reuse that causes issue #2.

## Test coverage gaps

The 14 `CombatBarkIntegrationTest` tests cover: taunt on first tick, no taunt on second tick, damage reaction (probability via seed search), death barks for two characters, victory bark on kill, heal action with bark, heal targeting lowest HP, heal cap at maxHp, and `Combatant.heal()` edge cases.

Not tested: exploration bark firing on sewer entry / enemy encounter / boss discovery, idle bark timer trigger after 30s, the double-bark-on-sewer-entry audio interruption, cooldown suppression of combat barks in rapid sequences, and `AndroidAudioPlayer` behavior (only `DesktopAudioPlayer` logic is exercised via `BarkAudioPlayerTest`'s fake).
