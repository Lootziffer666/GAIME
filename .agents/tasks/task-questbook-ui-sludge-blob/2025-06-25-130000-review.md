# Questbook UI polish and Sludge Blob attack rework

Two orthogonal features in one branch: Compose animation upgrades for the questbook overlay, pressure chip, and false quest markers on the UI side; and an `AttackStyle` enum on the combat side that gives the Sludge Blob a skip-tick ranged attack targeting random party members. The UI work is purely cosmetic (no behavioral test coverage required). The combat work is backed by six new deterministic tests using seeded `Random`.

Watch for:
- **Empty-party crash path** in `strikeRandomPartyMember`: `random.nextInt(living.size)` throws `IllegalArgumentException` when `living` is empty before `getOrNull` can guard it (confirmed).
- **ticksSinceLastAttack never resets** between encounters — if combatants are reused across fights, pacing carries over (confirmed).

## High-level view

The combat system gains a second attacking style via a new `AttackStyle` enum wired through `EnemyArchetype` → `Combatant` → `CombatEngine.enemyTurn()`. RANGED_SLOW enemies (currently only Sludge Blob) attack every other tick and select their target randomly from living party members instead of always striking the tank. The skip-tick pacing uses a mutable `ticksSinceLastAttack` counter on `Combatant`, which is never reset — not a problem in the current spawn-per-encounter lifecycle, but a latent issue if combatants are ever pooled.

The `strikeRandomPartyMember` method duplicates bark-emission logic from `strikeParty`. Both methods independently check death/damage bark thresholds with identical 40% probability gates. If that probability or the bark lookup changes, it must be changed in two places.

The UI changes use `AnimatedVisibility` with slide + fade for the questbook flash, `animateColorAsState` for pressure-chip color, and `rememberInfiniteTransition` driving jitter/flicker on false-marker chips. The `SliceQuestbookFlash` signature change from `(text: String)` to `(visible: Boolean, text: String)` enables exit animations that the old `?.let` pattern couldn't express.

## AttackStyle dispatch and tick pacing

`enemyTurn()` now branches on `enemy.attackStyle`. RANGED_SLOW checks `ticksSinceLastAttack % 2`: on even ticks the blob fires, on odd ticks it emits a "preparing to spit..." message. The counter increments unconditionally after the branch.

The counter lives on `Combatant` as a `var ticksSinceLastAttack: Int = 0` with no reset mechanism. In the current game, enemies are spawned fresh per encounter via `EnemyArchetype.spawn()`, so the counter always starts at zero. But the field is public-mutable with no documentation that fresh-spawn is the expected lifecycle. If an enemy is ever kept alive across encounters (retreating enemies, persistent world bosses), the pacing breaks silently.

```kotlin
// The dangerous path in strikeRandomPartyMember:
val living = livingParty()
val target = living.getOrNull(random.nextInt(living.size)) ?: return
```

When `living` is empty, `random.nextInt(0)` throws `IllegalArgumentException` before `getOrNull` executes. The equivalent code in `strikeParty` uses `firstOrNull()` which handles the empty case cleanly. The fix is a one-liner: `if (living.isEmpty()) return` before the `random.nextInt` call.

## Bark logic duplication between strike methods

`strikeRandomPartyMember` reimplements the death-bark and 40%-chance-damage-bark block from `strikeParty`. Both use the same probability constant (0.4f), the same helper functions (`deathBarkFor`, `damageBarkFor`), and the same guard conditions. If a future change adjusts the bark probability or adds a new bark category (critical hit, low-HP plea), it must be applied in both places. Extracting a `emitBarkIfApplicable(target, dealt, events)` helper would eliminate this.

## Test coverage gaps

Not tested: the empty-living-party edge case in `strikeRandomPartyMember` (which currently crashes), and blob behavior when it's the sole surviving combatant in a fight where all party members die mid-round.
