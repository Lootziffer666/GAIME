# Questbook UI polish and Sludge Blob attack rework (v2)

Two orthogonal features in one branch: Compose animation upgrades for the questbook overlay, pressure chip, and false quest markers; and an `AttackStyle` enum giving the Sludge Blob a skip-tick ranged attack targeting random party members. All four issues from v1 are resolved ‚Äî the empty-party crash is guarded, bark logic is deduplicated into `emitBarkIfApplicable`, the lifecycle contract is documented, and two new tests cover the empty-party edge case. Eight tests total, all deterministic via seeded `Random`.

Watch for:
- **RANGED_SLOW dodge message asymmetry** ‚Äî when dodging, `strikeRandomPartyMember` emits `"<name>'s attack is dodged."` while `strikeParty` emits the same format; however the dodge return happens before the `livingParty()` call, meaning the tick counter still increments even on a dodged turn, advancing pacing. This is arguably correct ("preparing" doesn't care if the party dodged), but the skip-tick rhythm continues invisibly through dodges with no player feedback on off-ticks (likely).
- **`RAT_ACCOUNTANT` inherits default MELEE but is boss-controlled** ‚Äî the boss uses `BossController` and bypasses `attackStyle` dispatch entirely since it's handled separately in `enemyTurn`. The default MELEE on `RAT_ACCOUNTANT` is dead metadata; harmless, but could confuse a future reader who tries to change boss attack patterns via the enum (possible).

**Verdict**: APPROVED

## High-level view

The v1 crash path (`random.nextInt(0)` on empty party) is eliminated by an early-return guard in `strikeRandomPartyMember`. The bark logic that was previously copy-pasted across both strike methods now lives in a single `emitBarkIfApplicable` helper called from both sites ‚Äî the same death/40%-damage gates, single source of truth. The `ticksSinceLastAttack` counter's spawn-per-encounter contract is documented with a KDoc comment explicitly prohibiting combatant reuse across fights, which addresses the lifecycle concern without adding reset machinery.

The test suite grew from 6 to 8 cases. The two new tests verify the empty-party edge: one with a single 1-HP member killed on the first tick, another with two blobs and two 1-HP members to exercise the multi-attacker path where `livingParty()` empties mid-round. Both confirm `CombatResult.DEFEAT` without exceptions.

The UI additions (AnimatedVisibility flash, animateColorAsState pressure chip, glitchy false markers via `rememberInfiniteTransition`) are cosmetic and don't require behavioral test coverage ‚Äî they're compile-verified through the existing `./gradlew desktopTest` build gate.

## Tick counter and dodge interaction

In `enemyTurn`, the RANGED_SLOW branch increments `ticksSinceLastAttack` unconditionally after the attack-or-prepare decision:

```kotlin
AttackStyle.RANGED_SLOW -> {
    if (enemy.ticksSinceLastAttack % 2 == 0) {
        strikeRandomPartyMember(enemy, enemy.attackPower, dodging, events)
    } else {
        events += CombatEvent.Message("${enemy.name} is preparing to spit...")
    }
    enemy.ticksSinceLastAttack++
}
```

`strikeRandomPartyMember` returns early on dodge with a message, but the counter still advances in the caller. The practical result: if the player dodges on an attack tick (0, 2, 4‚Ä¶), the blob's attack is negated but the rhythm still advances ‚Äî the next tick will be a "preparing" tick regardless. This means dodging doesn't "waste" the blob's attack in a way that grants the player an extra preparation tick. Whether this is a bug or a feature depends on intended design: if the blob's rhythm is physics-like (independent of success), it's correct; if dodging is meant to desync the blob, the counter should only advance on non-dodged ticks.

No test currently exercises dodge + RANGED_SLOW interaction.

## Boss exclusion from AttackStyle dispatch

The `enemyTurn` loop filters with `it.isAlive && it !== boss`, so the `RAT_ACCOUNTANT` never enters the `when (enemy.attackStyle)` branch. Its `attackStyle` field (default MELEE) is never read at runtime. The boss's attack damage comes from `bossController.takeTurn()` ‚Üí `strikeParty()` path, which is unaffected by this PR. The field is technically wasted memory, and a future developer might mistakenly try to give the boss `RANGED_SLOW` via the enum expecting it to work.
