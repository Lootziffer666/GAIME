# KMP waitroom with state-machine-driven particle canvas

GAIME is a greenfield Kotlin Multiplatform project (Android + Desktop/JVM) implementing an "AI waitroom" — a playful visual distraction for users while background AI work completes. The architecture centers on a `GameStateMachine` that drives lifecycle transitions (Idle through RevealReady) and a `SceneEngine` that manages interactive canvas scenes (particle-based letter swarm, color-cycling sprite placeholder). The Canvas composable runs a `withFrameNanos` animation loop gated on state-machine state and Android lifecycle.

**Watch for:** scene animation freezes in Idle and RevealReady — the canvas is visible but dead until "Start Thinking" is pressed (confirmed); `ManualSignalSource` is wired but never collected via Flow — the UI bypasses it entirely and calls `stateMachine.transition()` directly (confirmed); `GameSessionState.Failed` exists but no event transitions into it (confirmed).

## High-level view

The state machine enforces the critical invariant that AI results are hidden until the game round ends (GameOver in Thinking stays in Thinking, only GameOver in ReadyButPlaying reveals). The machine lacks a transition into the `Failed` state — the state exists in the enum but no event leads to it, making it unreachable dead code. `AiWorkState` (Idle/Working/Finished) is similarly unused anywhere.

The canvas frame loop's `isActive` flag only activates during Thinking and ReadyButPlaying, meaning in Idle state before AI starts and in RevealReady after game ends, the canvas is visible but frozen. Particles sit at initial positions or freeze mid-motion. On desktop, `LetterSwarmScene` initializes particle layout from the first frame's canvas dimensions and never re-layouts on resize.

`ManualSignalSource` exposes a `Flow<GameEvent>` designed as the contract for real AI signal providers. `WaitroomScreen` never collects this flow — it calls the state machine directly. When a real signal source is connected, `MutableSharedFlow` with replay=0 will silently drop events emitted before the collector subscribes.

Test coverage is solid for the state machine (8 tests) and particle physics (4 tests). No tests for `SceneEngine`, `SpriteAnimator`, canvas lifecycle gating, or Flow-based event delivery.

<summary>Details</summary>

## State machine: dead Failed state and unconditional close

The `Failed` state in `GameSessionState` has no inbound transition. No event in `GameEvent` maps to it. If the intent is to handle AI errors or network failures, the plumbing is missing. Similarly, `AiWorkState` (Idle/Working/Finished) is declared but never referenced from any other file.

`PlayerClosedGame` transitions to `Closed` unconditionally — from any state. A player can close during Thinking (before AI finishes) and the machine doesn't record whether the AI result was lost or whether cleanup is needed.

## Canvas lifecycle and the Idle freeze

In `WaitroomScreen`, the derivation:

```kotlin
val sceneIsActive = lifecycleActive && (
    currentState == GameSessionState.Thinking ||
    currentState == GameSessionState.ReadyButPlaying
)
```

On app launch (state = Idle), the canvas is visible but frozen. The LetterSwarm scene never gets `update()` calls, so particles sit at their initial positions. After `GameOver` transitions to `RevealReady`, animation also stops — particles freeze mid-motion wherever they were on the last frame.

## Particle layout fragility on resize

`LetterSwarmScene` calls `initParticles()` inside `draw()` using `drawScope.size`. This is a one-shot initialization (guarded by `initialized` flag), so if the desktop window resizes after the first frame, particles remain laid out for the original dimensions. They can end up clustered in a corner or overflowing bounds. Resetting `initialized` on size change and re-running layout would fix this.

The pointer interaction radius is hardcoded at 120 pixels. The Curious personality doubles it (`pointerRadius * 2f`) for its attract range, creating an asymmetry where Curious particles react at 240px but Nervous particles only dodge at 120px.

## Signal source: replay gap

`MutableSharedFlow(extraBufferCapacity = 10)` with default `replay = 0` means any event emitted before the collector starts is lost. If the AI backend emits `AiFinishedThinking` during a brief period where the collector hasn't started (e.g., during recomposition), the event vanishes silently. This is the single most important thing to fix before wiring a real AI signal source.

## Test coverage gaps

**Not tested:** `SceneEngine` switching and registration, `SpriteAnimator` frame advancement and looping/non-looping behavior, `GameCanvas` lifecycle gating (animation stops when `isActive` flips), the interaction between `WaitroomScreen` state and scene activation, any Flow-based event delivery path through `AiSignalSource`.
