# NPC tavern dialogues with voice-line playback

This change adds interactive NPC conversations to the tavern phase of the vertical slice. The Barkeep and Patron can be addressed via E-key, triggering scripted dialogue lines with synchronized WAV audio playback. The implementation extends `DialogueLine` with an optional `audioPath`, adds a `playRawPath` method to `BarkAudioPlayer` for non-BarkEvent audio, introduces a `SlicePhase.NPC_DIALOGUE` state, and wires everything through `DialogueOverlay`'s new `LaunchedEffect`. The same diff bundles a significant combat-system expansion (heal action, taunt/damage/death/victory barks, idle timers, exploration barks) that constitutes roughly half the change volume.

Watch for: NPC bark registry entries are registered under party-character speakers (BRUGG/VELLUM) rather than NPC-specific speakers, which conflates the Barkeep's identity with the party member Brugg in the bark metadata (confirmed). The `DialogueOverlay` audio trigger has no cancellation on rapid line advancement â€” if a player clicks through dialogue faster than the WAV plays, correctness depends on Compose recomposition timing rather than explicit lifecycle control (likely). The combat expansion is substantial unrelated scope bundled with the NPC dialogue feature.

## High-level view

The dialogue model gains a single nullable field (`audioPath`) and `DialogueOverlay` plays it via `LaunchedEffect(currentIndex)`. The playback lifecycle is tied to composition rather than the dialogue state machine, which creates an edge case around rapid advancement.

`BarkAudioPlayer.playRawPath` provides a bypass for lines that aren't driven by `BarkEvent` â€” it stops current audio, clears state, and plays an arbitrary path. The Barkeep and Patron WAV files live under the party-character directories (`bark/brugg/`, `bark/vellum/`), reusing the voice-archetype mapping rather than introducing NPC-specific directories.

NPC interaction is modeled as a phase transition: `TAVERN` â†’ `NPC_DIALOGUE` â†’ `TAVERN`. Entity selection happens via `GridWorld.requestInteraction()` (adjacent-tile check), and the E-key handler routes to `tavernScene.onEntityInteraction`.

The combat system received a parallel expansion: a `Heal` action, taunt barks on combat start, damage-reaction barks (40% probability, seeded Random), death barks, victory barks, and an idle-bark timer for exploration phases. This work is coherent on its own but orthogonal to NPC dialogues.

## Audio playback lifecycle in DialogueOverlay

`DialogueOverlay` fires playback inside `LaunchedEffect(currentIndex)`:

```kotlin
LaunchedEffect(currentIndex) {
    line.audioPath?.let { path ->
        barkAudioPlayer?.playRawPath(path)
    }
}
```

`playRawPath` is fire-and-forget â€” `audioPlayer.play(path)` starts playback on the platform thread and returns immediately. There's no coroutine-level cancellation that stops the underlying audio clip. `playRawPath` does call `audioPlayer.stop()` before `audioPlayer.play(path)`, which mitigates overlap â€” but only if `onAdvance` triggers recomposition fast enough for the next `LaunchedEffect` to fire before the old clip progresses audibly. The correctness depends on composition timing rather than explicit lifecycle management. A safer pattern: call `barkAudioPlayer?.stop()` inside `advanceDialogue()` directly, guaranteeing the previous clip ends regardless of recomposition order.

## Speaker identity in bark registries

The three NPC barks are registered as:

```kotlin
def(BarkEvent.BARKEEP_SPEND_SOME_COIN, BRUGG, "Spend some coin or get out.", ...)
def(BarkEvent.BARKEEP_BEEN_PLAYING_IN_SEWERS, BRUGG, "Been playing in the sewers, have we?", ...)
def(BarkEvent.PATRON_HE_SURE_IS_SLOW, VELLUM, "He sure is slow for a four-armed bartender.", ...)
```

This is intentional for audio-path resolution (`BarkAudioRegistry.pathFor` uses `character.audioDirectory` to derive `bark/brugg/` or `bark/vellum/`), but `BarkDefinition.character` for these events reports a party member, not the NPC. If any future code uses this field to display a speaker name or filter by "who said this," these will be misattributed.

## Idle bark timer

A `LaunchedEffect(phase)` polls every second, checking whether `clock() - lastActivityTime >= 30_000L`. The timer resets on user interactions (`lastActivityTime = clock()` is sprinkled throughout key handlers, combat actions, and dialogue advancement).

The gap: phase transitions don't explicitly reset `lastActivityTime`. When the `LaunchedEffect(phase)` restarts (because `phase` changed), the while-loop begins checking immediately with the stale timestamp. Combat phases don't update `lastActivityTime` on exit, so after a 30-second boss fight the idle timer fires instantly upon returning to exploration. Adding `lastActivityTime = clock()` at phase-transition points would fix this.

## Combat system expansion (bundled scope)

The combat engine gained heal, taunt, damage-reaction, death, and victory barks. The seeded `Random` parameter on `CombatEngine` is a good testability pattern. The taunt and victory bark pools overlap (e.g., `NIB_IS_THAT_ALL_YOUVE_GOT` appears in both) â€” the same line can play at combat start and combat end, which may or may not be intended.

## Test coverage

22 new tests across two files cover the core mechanisms well: `DialogueLine.audioPath` semantics, `playRawPath` behavior, NPC bark registry entries, combat taunt/damage/death/victory/heal bark triggers.

Not tested: the `NPC_DIALOGUE` phase transition in `SliceScreen`, the idle bark timer, E-key interaction routing, and exploration barks on trigger/entity interactions. These are integration-level behaviors baked into the Composable with no state-machine extraction, making isolated testing impractical without Compose UI test infrastructure.
