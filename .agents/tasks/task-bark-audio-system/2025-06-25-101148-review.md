# Bark audio system: streaming playback for 572 voice lines

This change adds audible voice lines to the bark system. 572 WAV files (~211 MB) for three characters (Nib, Brugg, Vellum) are extracted from 7z archives into Compose Resources, a registry maps every BarkEvent to its WAV path, and a cross-platform AudioPlayer abstraction plays one file at a time with immediate-interrupt semantics. The BarkEvent enum grows from 20 to 59 entries covering combat taunts, damage reactions, healing, exploration, and idle chatter. Memory footprint stays low by design: only the currently playing WAV is loaded; all others remain on disk.

**Watch for:** DesktopAudioPlayer reads the entire WAV into a byte array before playback (confirmed) — files average ~370 KB so this is fine for barks, but the pattern would not scale to music tracks. The AndroidAudioPlayer writes each WAV to a temp file before MediaPlayer can play it (confirmed); this adds disk I/O latency on every bark. `currentBark` state in BarkAudioPlayer can drift from actual playback state because there is no completion callback (confirmed).

## High-level view

The streaming/on-demand design keeps memory at single-file scale despite 572 files totalling 211 MB. `play()` calls `stop()` first, which closes the previous Clip (Desktop) or releases the previous MediaPlayer (Android), so at most one WAV occupies memory at any given moment.

BarkAudioRegistry enforces completeness at init time via `require()`: if any BarkEvent lacks a WAV mapping the app crashes immediately. This makes "missing audio" a build-time/startup problem rather than a silent runtime gap.

The 39 new BarkEvents are flavor-only (FlavorText effect). None trigger gameplay-affecting effects, keeping the expansion safe from unintended side-effects on quest logic. The `when` in QuestbookProcessor remains exhaustive, so adding future events without handling them is a compile error.

SliceDirector integrates audio as a late-set optional property rather than a constructor parameter to avoid breaking the trailing-lambda call pattern used throughout existing code. Silent failure when unset is intentional for test/headless environments.

1. **currentBark state drift** — `BarkAudioPlayer.currentBark` is set on `playBark()` but never cleared on natural playback completion. Add a completion listener or poll-based invalidation so `currentBark` returns to null when audio finishes.
2. **Android temp-file I/O on every bark** — `AndroidAudioPlayer` writes WAV bytes to a temp file for every `play()` call. Consider using `AssetFileDescriptor` from `context.assets` directly (Compose Resources maps to assets on Android), which gives a seekable FD without temp-file overhead.

4. **211 MB binary assets in git history** — 572 WAV files committed directly to the repo. If these ever need to change or be removed, the history carries the weight forever. Consider Git LFS for binary audio assets.

## currentBark state vs actual playback

`BarkAudioPlayer` tracks which bark is "playing" via `currentBark`, but neither platform implementation signals back when playback completes. On Desktop, the `LineEvent.Type.STOP` listener closes the clip but never notifies `BarkAudioPlayer`. On Android, the `OnCompletionListener` releases the player but likewise tells nobody upstream.

This means `currentBark` remains set to the last-fired event indefinitely after audio finishes. `isPlaying()` delegates correctly to the platform (`clip.isRunning` / `mediaPlayer.isPlaying`), so interruption logic works — but any code that checks `currentBark` for UI or decision-making will see a stale value.

```kotlin
// BarkAudioPlayer.kt line 32
fun playBark(event: BarkEvent) {
    val path = BarkAudioRegistry.pathFor(event)
    audioPlayer.stop()
    currentBark = event  // never cleared on natural completion
    audioPlayer.play(path)
}
```

Fix: add an `onComplete: (() -> Unit)?` callback to `AudioPlayer.play()`, or have platform implementations invoke a completion hook that clears `currentBark`.

## Android temp-file write pattern

`AndroidAudioPlayer.play()` loads the entire WAV into a byte array, writes it to a temp file, then hands the file descriptor to MediaPlayer. This introduces disk I/O latency on every bark — on low-end Android devices with slow flash, this could produce audible delay between the game event and the voice line.

```kotlin
// AndroidAudioPlayer.kt lines 31-39
val bytes = this::class.java.classLoader
    ?.getResourceAsStream("files/$resourcePath")
    ?.readBytes() ?: return
val tmp = File.createTempFile("bark_audio_", ".wav")
tmp.writeBytes(bytes)
```

Compose Resources bundles `files/` content as Android assets. Using `context.assets.openFd("files/$resourcePath")` gives a seekable `AssetFileDescriptor` that MediaPlayer accepts directly via `setDataSource(fd.fileDescriptor, fd.startOffset, fd.length)`, eliminating the temp-file step. This requires passing a `Context` to the player.

## DesktopAudioPlayer close-path race

Two paths call `clip.close()`:

1. The `LineEvent.Type.STOP` listener (fires on natural completion, on the audio thread)
2. The `stop()` method (fires on interruption, on the calling thread)

```kotlin
clip.addLineListener { event ->
    if (event.type == LineEvent.Type.STOP) {
        clip.close()  // path 1
    }
}

override fun stop() {
    currentClip?.let { clip ->
        if (clip.isRunning) { clip.stop() }
        clip.close()  // path 2
    }
}
```

If `stop()` is called while a natural-completion `STOP` event is being dispatched on the audio thread, both paths execute `close()`. The javax.sound.sampled spec leaves double-close behavior implementation-defined; some JVMs log warnings or throw `IllegalStateException`. A guard (`if (!clip.isOpen) return` in the listener) would make this deterministic.

## Repository size: 211 MB of WAV in git

572 WAV files committed directly to the git object store. Clones, CI checkouts, and GitHub storage are permanently heavier — even if files are later deleted, they remain in history. For a game asset library of this size, Git LFS stores binary blobs externally so only the currently checked-out versions are downloaded.

Not blocking for a private repo with few contributors, but will compound as the audio library grows.
