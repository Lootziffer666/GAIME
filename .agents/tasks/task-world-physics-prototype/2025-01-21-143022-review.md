# World physics prototype: Snow, Footprints, Wind, Blood, Torch Light

Five per-tile weather systems land as pure-logic grids in `:core` (SnowGrid, BloodGrid, WindState, orchestrated by WeatherState) with two new GLSL shaders in `:game` (SnowFilter, BloodFilter) and a standalone demo scene that composites all five effects. The approach splits cleanly along the existing architecture: simulation is renderer-agnostic in `:core`, visual representation lives in `:game` shader filters following the established UniformBlock/BaseProgramProvider pattern, and verification uses the screenshot harness with deterministic frozen time values.

**Watch for:** The per-tile grid data never actually reaches the shaders — SnowFilter/BloodFilter use a single scalar `intensity` rather than sampling the spatial grid, so footprints and splatter positions are invisible to the GPU (confirmed). PhysicsTestScene creates a RainFilter that is never applied to any view (confirmed dead code). SnowGrid.refill mutates cells in-place while iterating, creating order-dependent results (confirmed).

## High-level view

The `:core` weather layer is a straightforward set of 2D float grids with per-cell mutation. WindState drifts direction via a sine oscillation, SnowGrid accumulates uniformly and offers footprint/refill mechanics, BloodGrid splatters with Manhattan-distance radius and fades linearly. WeatherState orchestrates all three from a single `tick(dt)`.

The two new GLSL shaders (SnowFilter at fixedLocation 11, BloodFilter at 12) follow the established pattern but neither reads spatial grid data — they operate on a single global intensity. The per-tile simulation in `:core` has no visual expression beyond an averaged scalar fed to the shader. The gap is acknowledged in the FEAT-002 spec ("uniform for a small tile-grid sample (8x8 floats packed into vec4 uniforms)") but was not implemented.

PhysicsTestScene composites all effects using nested containers (outer=Blood, middle=Snow, inner=Lighting). It simulates 10 seconds of weather ticks deterministically, stamps footprints, splatters blood, places torch lights, then freezes. The screenshot harness adds five capture functions that isolate individual systems and one composite.

Not tested: the round-trip from grid state to shader uniform, interaction between refill and accumulate running simultaneously in `tick()`, behavior of grids with dimensions 0 or 1.

<details>
<summary>Issues (5)</summary>

1. **Grid-to-shader gap** — SnowFilter and BloodFilter take a single scalar intensity; per-tile snow depths and blood positions computed in `:core` have no spatial representation in the shader. Either pass grid data as packed uniforms (as the spec describes) or document this as a known limitation for the prototype.
2. **Dead RainFilter in PhysicsTestScene** — A RainFilter is constructed and configured but never attached to any view. Remove it or apply it to avoid confusion.
3. **In-place refill iteration order dependency** — SnowGrid.refill reads neighbor values while mutating the same array in a single pass, so cells processed later see already-refilled neighbors. Use a snapshot or double-buffer if deterministic spatial behavior matters.
4. **Wind parameters unused in accumulation** — `windDx`/`windDy` are passed to `SnowGrid.accumulate` but ignored in the body. The uniform accumulation rate means wind has no spatial effect on snow distribution.
5. **SnowFilter wind uniform naming** — The uniform is called `u_Wind` and the Kotlin property is `windAngle`, but the shader uses it as a linear shear factor, not an angle. The naming suggests radians but the math treats it as a displacement coefficient.

## Grid-to-shader disconnect

The entire `:core` weather layer computes per-tile values (snow depth at each cell, blood level at each cell) but the shaders collapse this to a single float. In PhysicsTestScene, the average snow depth across all 104 tiles is computed and passed as `SnowFilter.intensity`:

```kotlin
var totalSnow = 0f
for (y in 0 until 8) {
    for (x in 0 until 13) {
        totalSnow += weatherState.snow[x, y]
    }
}
val avgSnowDepth = totalSnow / (13 * 8)
```

Footprints (cells at depth 0) have no visual distinction from surrounding cells — the shader renders uniform white everywhere at the average intensity. BloodFilter is analogous: a global `intensity` and `splatterSeed` produce a procedural pattern unrelated to where `BloodGrid.splatter()` was actually called. The per-tile grids are simulation-correct but visually unrepresented.

## Dead code in PhysicsTestScene

At the bottom of `PhysicsTestScene.sceneMain()`:

```kotlin
val rainFilter = RainFilter(
    intensity = 0.4f,
    windAngle = weatherState.wind.direction,
    time = 3.0f
)
```

Constructed but never attached. The comment acknowledges the problem ("Rain applied to outermost - but we already have blood there") but leaves the dead code and unused import in place.

## In-place refill creates iteration-order dependency

`SnowGrid.refill` iterates row by row, left to right, calling `surroundingAverage` which reads current array values — but earlier cells in the same pass may already have been increased:

```kotlin
for (y in 0 until height) {
    for (x in 0 until width) {
        val idx = y * width + x
        val current = data[idx]
        if (current < 1f) {
            val avg = surroundingAverage(x, y)
            if (current < avg) {
                data[idx] = (current + amount).coerceAtMost(avg)
            }
        }
    }
}
```

A cell at (1,0) sees the already-refilled value of (0,0) in its surrounding average, while (0,0) saw the pre-refill value of (1,0). Footprints refill faster from the top-left direction than from bottom-right. For the current prototype (refillRate=0 in demos, small grids in tests) this won't surface, but if directional-neutral refill matters later, a double-buffer approach is needed.

## Wind accumulation is spatially inert

`SnowGrid.accumulate` accepts `windDx` and `windDy` but the body ignores them:

```kotlin
fun accumulate(dt: Float, rate: Float, windDx: Float = 0f, windDy: Float = 0f) {
    val amount = rate * dt
    for (i in data.indices) {
        data[i] = (data[i] + amount).coerceAtMost(1f)
    }
}
```

WeatherState passes `wind.dx` and `wind.dy` on every tick. The wind drives the shader's visual particle drift but has no spatial effect on where snow accumulates in the simulation grid. Anyone expecting wind to create drift patterns (more snow on leeward tiles) will find the API suggests it but the implementation doesn't deliver.
