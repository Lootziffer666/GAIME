# Chapter 2: The Market of Mandatory Commerce

Full gameplay flow from market through forest, shrine puzzle, Tax Collector Badger boss, and questbook page 2 return. The implementation adds 16 BarkEvents, 10 SlicePhases, 2 enemy archetypes, 2 maps, a new boss controller, extended questbook reactions, and 32 tests. The structure closely mirrors the Chapter 1 patterns and all registries remain exhaustive via their init{} guards.

Watch for: The SliceScreen never transitions the director's room context to `ROOM_FOREST_SHRINE` or `ROOM_FOREST_BOSS` before firing context-sensitive barks, meaning the "Electrical Works Permit" and "Surrender Filing: Rejected" reactions will never trigger at runtime (confirmed). The tests pass because they construct the correct context directly rather than going through the SliceScreen flow.

## High-level view

The QuestbookProcessor's context-aware reactions for BRUGG_DROP_YOUR_WEAPONS and VELLUM_CALLS_FOR_LIGHTNING depend on room contexts (`ROOM_FOREST_BOSS`, `ROOM_FOREST_SHRINE`) that SliceScreen never sets. The director stays in `FOREST_CTX` (roomId = "forest") for the entire forest/shrine/boss sequence, so the interesting reactions silently degrade to generic flavor text at runtime.

The data layer (BarkEvent, BarkRegistry, BarkAudioRegistry, SlicePhase, EnemyArchetype) follows established patterns — every new enum value has corresponding registry entries, and the BossControllerInterface extraction keeps CombatEngine polymorphic over boss types without changing its behavior.

## Context gap in the forest-to-shrine-to-boss flow

The forest enters with `director.enterRoom(FOREST_CTX)` where `FOREST_CTX` has `roomId = RoomContext.ROOM_FOREST`. When the player hits the shrine trigger, the phase changes to `CHAPTER2_SHRINE` and `VELLUM_ELEMENTS_MINE_TO_COMMAND` fires — but the director's context remains "forest". When the player later activates the lightning bark button, `fireAndFlash(BarkEvent.VELLUM_CALLS_FOR_LIGHTNING)` passes through `director.fireBark()` which calls `questbook.process(bark, context)` where `context.roomId` is still "forest":

```kotlin
// QuestbookProcessor.kt
BarkEvent.VELLUM_CALLS_FOR_LIGHTNING ->
    if (ctx.roomId == RoomContext.ROOM_FOREST_SHRINE) {  // "forest_shrine"
        react("Electrical Works Permit: Granted (Safety Notice: None)", RevealHidden)
    } else {
        react("Lightning request: No valid conductor", FlavorText)  // <-- this path fires at runtime
    }
```

The same pattern repeats for the boss intro: `BRUGG_DROP_YOUR_WEAPONS` fires while context is still `ROOM_FOREST`, but the reaction checks for `ROOM_FOREST_BOSS`.

The fix is two `director.enterRoom()` calls:
1. In the `TRIGGER_FOREST_SHRINE` handler, before setting the phase to CHAPTER2_SHRINE
2. In the `TRIGGER_FOREST_BOSS` handler (and the `tax_badger` entity interaction path), before firing the bark

Note that `enterRoom` with the same `mapId` ("forest_trail") won't trigger the map-transition reset (pressure/markers clearing), so this is safe to add without side effects. The pipeline test `shrineToForestDoesNotResetMarkersSameMap` already verifies this invariant.

## Duplicate boss entry via trigger and entity

The forest places both a `TRIGGER_FOREST_BOSS` tile and a `tax_badger` entity at position (24, 18). Both the `onTrigger` and `onEntityInteraction` handlers produce the same boss intro sequence. If GridWorld fires both a trigger and an entity interaction for the same tile step, the boss intro could fire twice (likely). The trigger fires on movement; entity interaction fires on pressing E. Since these are different input paths, this is not a runtime collision — but the entity-interaction path doesn't include the trigger path's `BRUGG_DROP_YOUR_WEAPONS` bark-then-dialogue sequence correctly from a context perspective (same missing enterRoom issue). Both paths need the context fix.
